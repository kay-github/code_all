## 教训

### 1. "验证基准即数据源"胜过"更精确的自算"
用户拿东财 app 核对结果。自算复权哪怕更精确，与东财差 5~80bp 也会被当成"数据错了"。切到 f25 同源后数值逐位一致，质疑消失。**先问用户拿什么验收，再选数据源。**

### 2. 两位小数的"假并列"顾虑要用全市场实测证伪，不要凭直觉设防
v1.1 曾因"f25 两位小数会制造假并列"拒绝将其作为排名键。实测 5793 只：除 0.00%（停牌类，333 只）外最大并列桶仅 5 只，排名区间误差 ±0.03%，且现有 `tieCount/rankPosition` 机制本来就支持并列。为一个未量化的顾虑维持 5700 行、70 分钟的管线，代价远大于收益。

### 3. 运行时长本身就是可靠性维度
70 分钟串行任务 = 任何一环失败当天就没数据，重试一次再花 70 分钟；1 分钟任务可以一晚安排三个幂等窗口（18:35/19:35/21:05），失败自动兜底。**缩短单次运行时间比加固每个环节更有效。**

### 4. 单源直取必须配独立源哨兵，且哨兵要覆盖"最会出错的样本"
f25 直取省掉全量校验，但需防上游口径漂移。腾讯 qfq 四哨兵（100bp 容差）中特意放了两只年内分红股（茅台、宝丰能源）盯复权口径。上线首日哨兵就立功：拦下盘中数据（79.96% vs 收盘 81.07%）。

### 5. 实时源要有"结算闸门"，且盘中运行是预期状态不是故障
东财 f25 盘中是实时值。push 触发的白天运行把盘中数据当日终快照，哨兵拦下但表现为 CI 红。正确做法：校验 ≥90% 行情时间戳（f124）等于目标交易日，未结算则以 **no-op 成功**退出，把发布留给晚间窗口。区分"预期的不可发布"与"真失败"，否则告警噪音会淹没真问题。

### 6. Vercel Blob 弱 ETag 坑（v1.3 快照卡死的根因之一）
Blob 读取返回 `W/"..."` 弱 ETag，而 `ifMatch` 条件写只认强格式 → `current.json` 每次条件更新都 412（`STOCK_SNAPSHOT_CURRENT_CONFLICT`），发布成功但指针永不切换。修复：读取侧统一 `normalizeEtag` 剥掉 `W/` 前缀（`stockSnapshotBlobStore.js`）。**条件写异常时先怀疑 ETag 格式，再怀疑并发。**

### 7. 交换数据源时尽量复用既有质量闸门框架，而不是绕过
v2.0 没有新建校验体系，而是给 `buildStockSnapshot` 增加 `reported-ytd.v1` 方法论分支：复用覆盖率闸门、新股排除、发布器校验、envelope 流程。改动集中在"哪些源合法、是否要求复权审计"两个开关上，测试与回滚都容易。

### 8. 上游字段会缺，缺法要归类为业务语义而非质量事故
东财对刚上市新股常返回空 f26（上市日期）。按"数据质量问题"隔离会让整批 quality=warning、健康降级；按业务语义归类（无上市日期 = 无法证明年初基准 = `NEW_LISTING` 排除）则批次干净。**每个"脏数据"先问：它有没有一个诚实的业务解释？**

### 9. 新旧管线切换要考虑"最后一班旧任务"
新 workflow 推送前 24 秒，旧 Python 定时任务启动，54 分钟后用旧口径覆盖了刚发布的 reported 快照。切换调度器时应先禁用旧任务（或让新任务在同一提交里替换旧文件并确认没有 in-flight run），再发布新数据。

### 10. 杂项
- GitHub PAT 默认没有 `workflow` scope，推 `.github/workflows/` 会被拒；凭据库里多个 token 时 Git 用第一个匹配条目，顺序即优先级。
- `vercel env pull` 对 sensitive 变量返回空串——本地复演生产发布要么新建专用密钥（本项目用 `STOCK_PUBLISH_SECRET`），要么走 OIDC。
- 东财 `push2` 主机偶发 502，`push2delay` 是稳定备胡；curl 直连 `api.siliconflow.cn` 等部分主机在 Windows schannel 下需 `--ssl-revoke-best-effort`（Node 不受影响）。
- 全市场扫描分页上限 100 行/页，59 页 ~32 秒；分页途中 `total` 变化说明撞上数据更新，应整批重来（现有 fetcher 已处理）。
